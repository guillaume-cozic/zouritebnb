# Contract Testing (OpenAPI = source of truth)

Keep the API and the front in lock-step through a single contract: the OpenAPI
schema generated by API Platform at `API/openapi.json`. Both sides are pinned to
it — the back proves its responses still match the schema, the front proves its
types still match the schema. See `CONTRACT_TESTING.md` at the repo root for the
high-level overview.

## Input

`$ARGUMENTS` may be:
- a route URI (e.g. `/api/accommodations`) or an Output class → **add a provider-side
  contract assertion** to that route's E2E test(s);
- `regenerate` (or empty after an API change) → **refresh the contract artifacts**.

## The golden rule

**Any change to the API surface (a new field, a new route, a changed type) means the
contract is stale. Regenerate both artifacts and commit them with the change:**

```bash
# 1. back: re-export the contract from API Platform
docker exec -i api-php php bin/console api:openapi:export --output=openapi.json

# 2. front: regenerate the TS types from the contract
cd front && npm run gen:api-types
```

Both regenerations are **idempotent** — re-running with no API change yields no diff.
A CI step can therefore fail the build when either file is stale (regenerate, then
`git diff --exit-code -- API/openapi.json front/src/api/schema.ts`).

Never hand-edit `API/openapi.json` or `front/src/api/schema.ts`.

## Provider side (back) — responses honor the contract

E2E tests validate real API responses against the published schema via the
`App\Tests\E2e\AssertsOpenApiContract` trait. It uses `opis/json-schema`
(JSON Schema draft 2020-12 = the OpenAPI 3.1 dialect, so union types like
`["string", "null"]` validate correctly — the older `league/openapi-psr7-validator`
does **not** handle these and must not be used).

To add a check, capture the response and assert against the **templated** path:

```php
use App\Tests\E2e\AssertsOpenApiContract;

final class GetAccommodationTest extends AccommodationApiTestCase
{
    use AssertsOpenApiContract;

    public function testShouldGetAccommodation(): void
    {
        $id = $this->insertAccommodation('Beach Villa', 'Luxury beach villa', 320.0);

        $response = static::createClient()->request('GET', '/api/accommodations/'.$id);

        self::assertResponseIsSuccessful();
        $this->assertResponseMatchesOpenApiContract($response, 'GET', '/api/accommodations/{id}');
    }
}
```

Steps:
1. **Identify the route** from `$ARGUMENTS` (resolve an Output class to its
   `uriTemplate`s as in the `e2e-test` skill).
2. **Pick the E2E test(s)** that already exercise GET item / GET collection for that
   route — those return a body worth validating. Add `use AssertsOpenApiContract;`
   to the class and the assertion to each success test.
3. **Capture the response** returned by `request(...)` (assign it to `$response`).
4. **Pass the templated URI**, not the concrete one: `'/api/accommodations/{id}'`,
   never `'/api/accommodations/'.$id`.
5. **Run** `docker exec -i api-php bin/phpunit tests/E2e/<Slice>/`.

Rules:
- Validate operations that return a body (GET item, GET collection, POST create).
  204 No Content responses have nothing to validate — the trait passes them through
  but asserts the body is empty.
- If a response fails validation, the implementation drifted from the documented
  schema: regenerate `openapi.json` (golden rule) and re-run. A persistent failure
  is a real contract bug to fix in the Output/serialization layer.
- Keep the existing business assertions (`assertJsonContains`, counts…) — the
  contract assertion is **additive**, it checks shape, not values.

## Consumer side (front) — types match the contract

- `front/src/api/schema.ts` — types generated from the contract (`openapi-typescript`).
  Never edited by hand; regenerate with `npm run gen:api-types`.
- `front/src/api/contract.ts` — compile-time guard pinning hand-written domain types
  to the generated contract.

To pin a hand-written type to the contract, add a block to `contract.ts`:

```ts
import type { components } from './schema';
import type { AccommodationListItem } from '../features/homepage/HomepageTypes';

type Expect<T extends true> = T;
type IsNever<T> = [T] extends [never] ? true : false;
type FieldsCompatible<Front, Contract> = {
  [K in keyof Front & keyof Contract]: NonNullable<Front[K]> extends NonNullable<Contract[K]>
    ? true : false;
}[keyof Front & keyof Contract] extends true ? true : false;

type Schema = components['schemas']['AccommodationEntity.jsonld-accommodation.list'];

// 1. every field the front relies on must still exist in the contract
type _KeysExist = Expect<IsNever<Exclude<keyof AccommodationListItem, keyof Schema>>>;
// 2. each shared field's type must be compatible with the contract
type _TypesMatch = Expect<FieldsCompatible<AccommodationListItem, Schema>>;
```

The guard asserts the front type is a **valid subset** of what the contract permits
(the consumer direction): the front must never expect a field the contract forbids,
nor a type narrower than allowed. This tolerates API Platform's over-nullable output
(e.g. `(string | null)[]` for an array) without false positives, while still failing
on a removed/renamed field or an incompatible type change.

Steps:
1. **Find the schema name** in `schema.ts` for the relevant resource + serialization
   group (e.g. `AccommodationEntity.jsonld-accommodation.list` vs `…-accommodation.read`).
2. **Add a `_KeysExist` + `_TypesMatch` block** for each hand-written type to pin.
3. **Run** `cd front && npm run typecheck`. A drift surfaces as
   `Type 'false' does not satisfy the constraint 'true'`.

Rules:
- The guard is enforced by `npm run typecheck` (`tsc --noEmit`, scans every file).
  CRA's `build`/`test` only type-check imported files, so `typecheck` is the gate —
  wire it into CI.
- `contract.ts` has no runtime behaviour; it exists purely for the type checker.
- Prefer the matching serialization group: a list type → the `.list` schema, a detail
  type → the `.read` schema.

## End-to-end checklist when the API changes

1. Update the back (Output class, provider, processor…).
2. `docker exec -i api-php php bin/console api:openapi:export --output=openapi.json`
3. `cd front && npm run gen:api-types`
4. `docker exec -i api-php bin/phpunit tests/E2e` (provider contract holds)
5. `cd front && npm run typecheck` (consumer contract holds)
6. Commit code + `openapi.json` + `schema.ts` together.
