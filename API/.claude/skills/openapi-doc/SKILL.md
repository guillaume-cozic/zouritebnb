# OpenAPI Documentation Enrichment for API Platform Routes

Enrich the OpenAPI specification generated by API Platform for a given resource: add descriptions, examples, summaries, and request body examples so the Swagger UI is exploitable by API consumers.

## Input

The user provides a resource to document: `$ARGUMENTS`

This can be a file path to the Output class, a FQCN, or a route URI (e.g. `/api/accommodations`). Resolve it to the Output class that holds the `#[ApiResource]` attribute.

## Steps

1. **Read the Output class** to list all declared operations (Get, GetCollection, Post, Patch, Put, Delete), their `uriTemplate`, `input`, and serialization groups.
2. **Read the Input class(es)** to understand which fields each write operation accepts and their types.
3. **Read the Doctrine entity** (from `stateOptions`) to understand field types and which fields map to domain concepts.
4. **Read the domain entity** (`src/<Slice>/Domain/Entity/`) and **domain exceptions** to understand validation rules (what makes a request invalid).
5. **Add `#[ApiProperty]` attributes** on every property of the Output class and each Input class.
6. **Add `openapi:` parameter** on every operation in the Output class `#[ApiResource]` attribute.
7. **Verify** the OpenAPI export with `docker exec -T api-php bin/console api:openapi:export --yaml 2>&1 | head -150`.

## What to enrich

### Properties — `#[ApiProperty]` on Output and Input classes

Add `#[ApiProperty(description: '...', example: ...)]` on every public property that has a `#[Groups]` attribute.

```php
use ApiPlatform\Metadata\ApiProperty;

#[Groups(['<slice>:read'])]
#[ApiProperty(description: 'Identifiant unique (UUID)', example: '01961e2f-dead-7000-beef-000000000001')]
public ?string $id = null;
```

Guidelines for descriptions and examples:
- **`id`** → `description: 'Identifiant unique (UUID)'`, `example:` a realistic UUID v7
- **string fields** → describe the field purpose, give a realistic example
- **float/int fields** → describe the field purpose and constraints (e.g. "strictly positive"), example is a realistic numeric value
- **boolean fields** → describe what true/false means
- **date fields** → example in ISO 8601 format
- Write descriptions in French, matching the domain language

### Operations — `openapi:` parameter on each operation

Add an `openapi:` named argument with `new OpenApiOperation(summary: '...', description: '...')` on every operation.

#### Read operations (Get, GetCollection)

Only need `summary` and `description`:

```php
new Get(
    uriTemplate: '/<slices>/{id}',
    openapi: new OpenApiOperation(
        summary: 'Récupérer un <slice>',
        description: 'Retourne le détail complet d\'un <slice> par son identifiant UUID.',
    ),
    normalizationContext: ['groups' => ['<slice>:read']],
),
new GetCollection(
    uriTemplate: '/<slices>',
    openapi: new OpenApiOperation(
        summary: 'Lister les <slices>',
        description: 'Retourne la liste paginée de tous les <slices> disponibles.',
    ),
    normalizationContext: ['groups' => ['<slice>:list']],
),
```

#### Write operations (Post, Patch, Put, Delete)

Add `summary`, `description`, and `requestBody` with named examples (valid + invalid cases derived from domain validation rules):

```php
new Post(
    uriTemplate: '/<slices>',
    status: 201,
    openapi: new OpenApiOperation(
        summary: 'Créer un <slice>',
        description: 'Crée un nouveau <slice>. <Validation constraints in prose>.',
        requestBody: new RequestBody(
            content: new \ArrayObject([
                'application/ld+json' => new MediaType(
                    examples: new \ArrayObject([
                        'valid' => new Example(
                            summary: 'Requête valide',
                            value: [/* all fields with valid values */],
                        ),
                        '<invalid_case_key>' => new Example(
                            summary: 'Invalide : <short reason>',
                            description: '<What error is returned and why>.',
                            value: [/* payload that triggers domain error */],
                        ),
                        // one Example per known validation rule
                    ]),
                ),
            ]),
        ),
    ),
    denormalizationContext: ['groups' => ['<slice>:write']],
    input: <Action><Slice>Input::class,
    processor: <Action><Slice>Processor::class,
),
```

For PATCH/PUT, use `'application/merge-patch+json'` as the content type key instead of `'application/ld+json'`.

For DELETE, no `requestBody` — only `summary` and `description`.

## Imports

The Output class needs these imports for the OpenAPI enrichment:

```php
use ApiPlatform\Metadata\ApiProperty;
use ApiPlatform\OpenApi\Model\Example;
use ApiPlatform\OpenApi\Model\MediaType;
use ApiPlatform\OpenApi\Model\Operation as OpenApiOperation;
use ApiPlatform\OpenApi\Model\RequestBody;
```

The Input class only needs:

```php
use ApiPlatform\Metadata\ApiProperty;
```

Always alias `Operation` as `OpenApiOperation` to avoid confusion with `ApiPlatform\Metadata\Operation`.

## Parameter ordering in operations

When adding `openapi:` to an operation, place it **after** `uriTemplate` and `status` but **before** `normalizationContext`, `denormalizationContext`, `input`, and `processor`:

```php
new Post(
    uriTemplate: '/<slices>',
    status: 201,
    openapi: new OpenApiOperation(...),     // here
    denormalizationContext: ['groups' => [...]],
    input: ...,
    processor: ...,
),
```

## Rules

- `declare(strict_types=1)` in every file
- Write descriptions and summaries in French
- The `summary` is a short verb phrase (imperative): "Créer un hébergement", "Lister les hébergements"
- The `description` is a full sentence explaining what the route does and any constraints
- Include one `'valid'` example and one example per known domain validation error for write operations
- Invalid examples should show what field is missing or what value triggers the error
- Invalid example `description` should state what HTTP status and error is returned
- Do not modify the `shortName`, `provider`, `stateOptions`, or the `fromEntity()` method
- Do not modify serialization groups, processors, or input class references
- Only add imports that are actually used
